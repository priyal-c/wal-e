# WAL-E Access Requirements Guide
## For SAs and Customers

---

> **TL;DR:** WAL-E needs a short-lived **Personal Access Token (PAT)** with **workspace admin read access**. It makes **100% read-only** API calls. No data is modified, no clusters are started, no jobs are triggered.

## 1. Overview

WAL-E (Well-Architected Lakehouse Evaluator) is an automated assessment tool that evaluates a Databricks workspace against the [Well-Architected Lakehouse Framework](https://docs.databricks.com/aws/en/lakehouse-architecture/well-architected). It connects via the Databricks CLI and REST API to collect workspace metadata, score it against 99 best practices across 7 pillars, and generate a readout deck.

**Key facts for the customer:**
- All API calls are **read-only** (HTTP GET only)
- **No data is read** from tables, volumes, or storage -- only metadata and configuration
- **No resources are created, modified, or deleted**
- The assessment token can be set to **expire in 1 day**
- A full **audit trail** of every API call is provided as a deliverable

## 2. What WAL-E Collects (and Doesn't)

### What IS Collected (Metadata Only)

| Category | What's Read | Example |
|----------|------------|---------|
| **Identity** | Current user info, group memberships | User email, group names |
| **Unity Catalog** | Metastore config, catalog names & owners, external locations, storage credentials | Catalog count, isolation modes |
| **Compute** | Cluster names & states, SQL warehouse configs, cluster policies, instance pools | Running cluster count, warehouse sizes |
| **Security** | Workspace settings (DBFS browser, export, token lifetime), IP access lists | Config flags (true/false) |
| **Operations** | Job names, pipeline states, serving endpoints, git repos, init scripts, secret scope names, group names | Job count, pipeline failure states |
| **Workspace** | Root-level directory listing (names & types only) | Folder names, notebook counts |

### What is NOT Collected

| NOT Collected | Why |
|---------------|-----|
| Table data / row content | WAL-E never reads actual data from tables |
| File/volume contents | No files or volumes are accessed |
| Secret values | Only scope names are listed, never secret values |
| Notebook code | No notebook content is read |
| Query text / SQL statements | No query history content is accessed |
| Credentials / tokens of other users | Only the assessment token's own metadata |
| Personal data of users | Only usernames/emails from group membership |

## 4. Step-by-Step Setup for Customer Sessions

### Step 1: Customer Generates a PAT Token

Share these instructions with the customer:

```
1. Log in to your Databricks workspace as a WORKSPACE ADMIN
2. Click your username (top-right) > "Settings"
3. Go to "Developer" > "Access tokens"
4. Click "Generate New Token"
5. Description: "WAL-E Assessment - [SA Name] - [Date]"
6. Lifetime: 1 day (24 hours) -- the assessment takes ~15 minutes
7. Click "Generate"
8. COPY the token immediately (it won't be shown again)
9. Share the token securely with the SA (e.g., via encrypted channel, NOT email)
```

**Important:** The token inherits the permissions of the user who creates it. For a complete assessment, the token should be created by a **workspace admin** who also has **metastore admin** access.

### Step 2: SA Configures the CLI Profile

```bash
# Create a dedicated CLI profile for this assessment
databricks configure --profile customer-assessment --host https://CUSTOMER-WORKSPACE-URL --token

# When prompted, paste the customer's PAT token

# Verify the connection works
wal-e validate --profile customer-assessment
```

### Step 3: Run the Assessment

```bash
# Run in interactive mode (customer can watch each step)
wal-e assess --profile customer-assessment --interactive

# Or run a quick scan
wal-e assess --profile customer-assessment --output ./customer-assessment-results
```

### Step 4: Clean Up After Assessment

```bash
# Remove the CLI profile after the assessment
# Edit ~/.databrickscfg and remove the [customer-assessment] section

# The customer's PAT token auto-expires after 1 day
# Customer can also manually revoke it immediately:
# Settings > Developer > Access tokens > Revoke
```

## 5. Required Permissions by Collector

WAL-E runs 6 collectors. Here is exactly what each one needs:

### Collector 1: Authentication & Identity

| Permission Level | Required | Fallback if Missing |
|-----------------|----------|---------------------|
| **CAN_VIEW** on workspace | Yes | Assessment cannot start |

| API Call | Databricks Permission | Admin Required? |
|----------|----------------------|-----------------|
| `databricks auth describe` | Any authenticated user | No |
| `databricks current-user me` | Any authenticated user | No |

### Collector 2: Governance (Unity Catalog)

| Permission Level | Required | Fallback if Missing |
|-----------------|----------|---------------------|
| **Metastore admin** or **USE CATALOG** on catalogs | Recommended | Partial catalog list returned |

| API Call | Databricks Permission | Admin Required? |
|----------|----------------------|-----------------|
| `GET /api/2.1/unity-catalog/metastore_summary` | Metastore user | No (but admin for full details) |
| `GET /api/2.1/unity-catalog/catalogs` | USE CATALOG or metastore admin | Metastore admin for ALL catalogs |
| `GET /api/2.1/unity-catalog/external-locations` | Metastore admin | Yes (for complete list) |
| `GET /api/2.1/unity-catalog/storage-credentials` | Metastore admin | Yes (for complete list) |

### Collector 3: Compute

| Permission Level | Required | Fallback if Missing |
|-----------------|----------|---------------------|
| **CAN_MANAGE** or workspace admin | Recommended | Only user's own clusters shown |

| API Call | Databricks Permission | Admin Required? |
|----------|----------------------|-----------------|
| `GET /api/2.1/clusters/list` | CAN_ATTACH_TO or admin | Admin for ALL clusters |
| `GET /api/2.0/sql/warehouses` | CAN_USE or admin | Admin for ALL warehouses |
| `GET /api/2.0/cluster-policies/list` | Any workspace user | No |
| `GET /api/2.0/instance-pools/list` | CAN_ATTACH_TO or admin | Admin for ALL pools |

### Collector 4: Security

| Permission Level | Required | Fallback if Missing |
|-----------------|----------|---------------------|
| **Workspace admin** | Yes | Security settings cannot be read |

| API Call | Databricks Permission | Admin Required? |
|----------|----------------------|-----------------|
| `GET /api/2.0/workspace-conf` | Workspace admin | **Yes** |
| `GET /api/2.0/ip-access-lists` | Workspace admin | **Yes** |
| `GET /api/2.0/token/list` | Any authenticated user (own tokens) | No |

### Collector 5: Operations

| Permission Level | Required | Fallback if Missing |
|-----------------|----------|---------------------|
| **CAN_VIEW** on jobs/pipelines or workspace admin | Recommended | Partial job/pipeline list |

| API Call | Databricks Permission | Admin Required? |
|----------|----------------------|-----------------|
| `GET /api/2.1/jobs/list` | CAN_VIEW or admin | Admin for ALL jobs |
| `GET /api/2.0/pipelines` | CAN_VIEW or admin | Admin for ALL pipelines |
| `GET /api/2.0/serving-endpoints` | CAN_QUERY or admin | Admin for ALL endpoints |
| `GET /api/2.0/repos` | CAN_READ or admin | Admin for ALL repos |
| `GET /api/2.0/global-init-scripts` | Workspace admin | **Yes** |
| `GET /api/2.0/groups/list` | Any workspace user | No |
| `GET /api/2.0/secrets/list-scopes` | Any workspace user | No |

### Collector 6: Workspace Structure

| Permission Level | Required | Fallback if Missing |
|-----------------|----------|---------------------|
| **CAN_READ** on workspace root | Recommended | Partial listing |

| API Call | Databricks Permission | Admin Required? |
|----------|----------------------|-----------------|
| `GET /api/2.0/workspace/list?path=%2F` | CAN_READ or admin | Admin for complete listing |

## 6. Complete API Endpoint Reference

Below is every API endpoint WAL-E calls, organized by HTTP method. **All calls are GET (read-only).**

```
# Authentication (2 calls)
databricks auth describe
databricks current-user me

# Unity Catalog Governance (4 calls)
GET /api/2.1/unity-catalog/metastore_summary
GET /api/2.1/unity-catalog/catalogs
GET /api/2.1/unity-catalog/external-locations
GET /api/2.1/unity-catalog/storage-credentials

# Compute Infrastructure (4 calls)
GET /api/2.1/clusters/list
GET /api/2.0/sql/warehouses
GET /api/2.0/cluster-policies/list
GET /api/2.0/instance-pools/list

# Security Configuration (3 calls)
GET /api/2.0/workspace-conf?keys=enableResultsDownloading,enableDbfsFileBrowser,enableExportNotebook,maxTokenLifetimeDays,enableIpAccessLists
GET /api/2.0/ip-access-lists
GET /api/2.0/token/list

# Operations (7 calls)
GET /api/2.1/jobs/list
GET /api/2.0/pipelines
GET /api/2.0/serving-endpoints
GET /api/2.0/repos
GET /api/2.0/global-init-scripts
GET /api/2.0/groups/list
GET /api/2.0/secrets/list-scopes

# Workspace Structure (1 call)
GET /api/2.0/workspace/list?path=%2F

# ──────────────────────────
# Total: 21 API calls
# All: HTTP GET (read-only)
# Write calls: 0 (zero)
```

## 7. Optional: System Tables Access for Deep Assessment

For a **deeper assessment** (cost analysis, query performance, audit trail analysis), WAL-E can optionally run SQL queries against [Databricks System Tables](https://docs.databricks.com/en/administration-guide/system-tables/index.html). This requires:

| Requirement | Details |
|-------------|---------|
| **System tables enabled** | Account admin must enable system tables |
| **SQL Warehouse** | A running SQL warehouse to execute queries |
| **SELECT on system schema** | `GRANT SELECT ON SCHEMA system.billing TO <user>` (and similar for other system schemas) |

### System Tables Used (If Available)

| System Table | Purpose | Pillar |
|-------------|---------|--------|
| `system.billing.usage` | DBU consumption & cost trends | Cost Optimization |
| `system.compute.clusters` | Cluster lifecycle & usage patterns | Cost, Performance |
| `system.query.history` | Query performance & failure rates | Performance |
| `system.access.audit` | Security audit events | Security |
| `system.information_schema.tables` | Table metadata & storage analysis | Governance, Performance |
| `system.lakeflow.job_run_timeline` | Job success/failure rates | Reliability, Operations |
| `system.access.table_lineage` | Data lineage tracking | Governance |

### Granting System Table Access

The customer's account admin can run:

```sql
-- Grant billing usage access
GRANT SELECT ON SCHEMA system.billing TO `assessment-user@company.com`;

-- Grant compute monitoring access
GRANT SELECT ON SCHEMA system.compute TO `assessment-user@company.com`;

-- Grant query history access
GRANT SELECT ON SCHEMA system.query TO `assessment-user@company.com`;

-- Grant audit log access
GRANT SELECT ON SCHEMA system.access TO `assessment-user@company.com`;

-- Grant information schema access (usually available by default)
GRANT SELECT ON SCHEMA system.information_schema TO `assessment-user@company.com`;
```

**Note:** System table access is **optional**. WAL-E produces a complete assessment using only the REST API calls listed in Section 6. System tables provide supplementary depth for cost and performance analysis.

## 8. Security & Privacy Assurances

Share these assurances with customers who have security concerns:

### Data Handling

| Concern | Assurance |
|---------|-----------|
| **Data at rest** | Assessment results are stored locally on the SA's machine only |
| **Data in transit** | All API calls use HTTPS/TLS encryption |
| **Data retention** | The SA deletes assessment files after delivering the readout |
| **PII exposure** | Only workspace user emails and group names are collected (for governance scoring) |
| **Credential storage** | The PAT token is stored in `~/.databrickscfg` and deleted after assessment |

### What WAL-E Will NEVER Do

- **Never** read table data, row content, or query results
- **Never** execute any notebook or job
- **Never** create, modify, or delete any workspace resource
- **Never** start or stop any cluster or warehouse
- **Never** modify any configuration or setting
- **Never** access secret values (only scope names)
- **Never** transmit data to any external service or third party
- **Never** store credentials beyond the local CLI profile

### Audit Trail

WAL-E generates a complete `WAL_Assessment_Audit_Report.md` that documents:
- Every API call made (command, endpoint, timestamp, duration)
- Every piece of raw output received
- What evidence was used for which finding

This audit report can be shared with the customer's security team for review.

## 9. Troubleshooting Access Issues

### Common Issues and Solutions

| Symptom | Cause | Solution |
|---------|-------|----------|
| `401 Unauthorized` | Token expired or invalid | Customer regenerates PAT token |
| `403 Forbidden` on workspace-conf | User is not workspace admin | Ask customer to use admin account |
| `403 Forbidden` on catalogs | User lacks metastore admin | Ask customer to grant metastore admin or USE CATALOG |
| Empty cluster list | User lacks CAN_ATTACH_TO | Ask customer to use admin account |
| Empty job list | User lacks CAN_VIEW on jobs | Ask customer to use admin account |
| `Connection refused` | Network/firewall blocking | Verify workspace URL; check VPN/proxy settings |
| `SSL certificate error` | Corporate proxy intercepting | Set `REQUESTS_CA_BUNDLE` or use `--insecure` flag |
| Partial catalog count | User only sees permitted catalogs | Metastore admin needed for complete count |

### Minimum vs. Recommended Access

| Access Level | What You Get | Coverage |
|-------------|-------------|----------|
| **Regular user** | Own clusters, permitted catalogs, own jobs | ~40% of best practices scored |
| **Workspace admin** | All clusters, warehouses, security config, all jobs | ~80% of best practices scored |
| **Workspace admin + Metastore admin** | All of above + all catalogs, credentials, locations | ~95% of best practices scored |
| **Above + System tables** | Full above + billing, audit, query history | **100% of best practices scored** |

### Validation Command

Before running the assessment, always validate access:

```bash
# Quick connectivity check
wal-e validate --profile customer-assessment

# Expected output for full access:
# ✓ Connected to https://customer-workspace.cloud.databricks.com
# ✓ Authenticated as: admin@company.com
# ✓ Workspace admin: Yes
# ✓ Unity Catalog access: Yes
# ✓ Metastore admin: Yes
```

### WAL-E Assessment: What We Need From You

**Hi [Customer Name],**

We'll be running a Well-Architected Lakehouse Assessment on your Databricks workspace. Here's what we need:

**1. A Personal Access Token (PAT)**

**What the tool does NOT do:**
- Does NOT read any table data or file contents
- Does NOT create, modify, or delete anything
- Does NOT start any clusters or run any jobs
- Does NOT access secret values

**The assessment takes approximately 15 minutes.** We'll walk you through the results and provide a complete audit trail of all API calls made.

**After the assessment:** You can revoke the token immediately in Settings > Developer > Access tokens.
